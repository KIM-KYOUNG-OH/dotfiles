## 2. RFP 신규구축 (02_RFP_신규구축) 검토

### 2.1 개선 필요 항목

#### A. 프로젝트 개요 섹션
| 현재 | 개선 권고 |
|------|-----------|
| 사업 목적이 추상적 | KPI 포함 필요 (예: "입고 처리 시간 30% 단축", "출고 정확도 99.9% 달성") |
| 예상 개발 기간 "12~18개월"만 명시 | Phase별 마일스톤 제시 권고 |

#### B. 요구사항 명세
| 항목 | 문제점 | 개선 권고 |
|------|--------|-----------|
| 성능 요구사항 | "일일 10,000건 이상" - 피크 시간대 처리량 미정의 | 시간당 최대 처리량, 피크타임 동시 트랜잭션 수 추가 |
| 동시 사용자 | "PC 50명, PDA 100대" - 최대치인지 평균인지 불명확 | 평균/피크 구분하여 명시 |
| 가용성 99.5% | 다운타임 허용 시간대 미정의 | 비즈니스 시간(09-18시) vs 24시간 구분 필요 |

#### C. 기술 요구사항
| 항목 | 문제점 | 개선 권고 |
|------|--------|-----------|
| 아키텍처 | "Microservices 또는 Modular Monolith" - 선택권 과다 | 선호 아키텍처 명시 또는 제안사 근거 요청 |
| 인프라 | AWS만 명시, 상세 [TBD] | VPC 구성, Region, 보안그룹 등 기본 요구사항 추가 |
| DR(재해복구) | 언급 없음 | DR 전략, RTO/RPO 목표 추가 필요 |

#### D. 외부 연동 요구사항
| 항목 | 문제점 | 개선 권고 |
|------|--------|-----------|
| 케이타운포유 연동 | API 스펙/문서 참조 없음 | 연동 API 문서 첨부 또는 접근 방법 명시 |
| 운송사 API | 각 운송사별 연동 복잡도 미언급 | 난이도/우선순위 명시 권고 |
| 자동화 설비 | 프로토콜 정보 부족 | 카톤랩, 제네시스 등 통신 프로토콜 명시 |

#### E. 평가 기준
| 현재 | 문제점 | 개선 권고 |
|------|--------|-----------|
| 기술 능력 30점 | 세부 평가 항목 없음 | 아키텍처 설계 10점, 연동 경험 10점, 기술 스택 적합성 10점 등 세분화 |
| 비용 20점 | 평가 방식 미정의 | 최저가 대비 점수화 방식 명시 필요 |

#### F. 누락된 섹션
- **데이터 마이그레이션 계획** - 기존 데이터 존재 시 이관 방안
- **교육 및 인수인계 요구사항** - 개발 완료 후 운영 전환
- **문서화 요구사항** - 산출물 목록 명시 필요
- **테스트 요구사항** - UAT, 성능테스트, 보안테스트 기준
- **변경 관리 프로세스** - 요구사항 변경 시 절차

## 3. RFP 운영유지보수 (03_RFP_운영유지보수) 검토

### 3.1 개선 필요 항목

#### A. 현재 시스템 현황
| 항목 | 문제점 | 개선 권고 |
|------|--------|-----------|
| 코드 규모 | 미언급 | LOC, 파일 수, 모듈 수 등 규모 정보 제공 |
| 기술 부채 | 미언급 | 알려진 이슈, 기술 부채 목록 제공 |
| 현재 인프라 | 상세 정보 없음 | 서버 구성, DB 용량, 트래픽 현황 제공 |

#### B. SLA 요구사항
| 항목 | 문제점 | 개선 권고 |
|------|--------|-----------|
| 장애 등급 정의 | "주요 기능"의 범위 불명확 | 기능별 중요도 등급 명시 (출고 > 입고 > 재고 등) |
| 대응 시간 | 업무시간/비업무시간 구분 없음 | 평일 업무시간, 주말, 공휴일별 SLA 차등화 |
| 페널티 | 미언급 | SLA 미준수 시 패널티 조항 추가 권고 |

#### C. 운영 범위
| 항목 | 문제점 | 개선 권고 |
|------|--------|-----------|
| 지원 시간 | "평일 09:00~18:00"만 명시 | 긴급 장애 시 24시간 대응 범위 정의 |
| 현장 지원 | "필요시"로 모호함 | 월 N회 정기 방문 등 구체화 |
| 보고 체계 | 리포트 항목만 있음 | 보고 대상, 회의 주기 명시 |

#### D. 추가 개발 범위
| 항목 | 문제점 | 개선 권고 |
|------|--------|-----------|
| 기능별 우선순위 | 비즈니스 가치/긴급도 근거 없음 | 각 기능의 비즈니스 임팩트 설명 추가 |
| 일정 | "3개월", "6~12개월"만 명시 | 의존관계, 선행 조건 명시 |
| 범위 변경 | 프로세스 없음 | 추가 개발 요청 시 절차 명시 |

#### E. 비용 구조
| 현재 | 문제점 | 개선 권고 |
|------|--------|-----------|
| 월간 비용만 요청 | 상주/원격 구분 없음 | 상주 인력 비용, 원격 지원 비용 분리 |
| M/M 단가 요청 | 직급별 단가 가이드 없음 | PM, PL, SR, Jr 등 직급별 가이드라인 제시 |
| 추가 개발 비용 | 별도 | 운영과 개발 비용 통합 옵션도 요청 권고 |

#### F. 누락된 섹션
- **인수인계 계획** - 현재 개발팀 → 운영팀 전환 방안
- **소스코드 접근권한** - 저장소 접근, 배포 권한 범위
- **개발환경 제공** - 운영사에 개발/테스트 환경 제공 여부
- **보안 심사** - 운영사 보안 자격 요건 (ISMS 등)

## Uncertainty Map

**낮은 확신도 영역**:
- 실제 비즈니스 상황과 예산 제약을 모르므로, 일부 권고사항은 현실과 맞지 않을 수 있음
- 운송사 API 연동 복잡도는 실제 API 문서 없이 판단 어려움
- 자동화 설비(카톤랩, 제네시스) 연동 난이도는 실제 프로토콜 문서 필요
- 케이타운포유 연동 범위와 복잡도는 API 스펙에 따라 달라질 수 있음

**의견이 달라질 수 있는 부분**:
- 예상 예산 공개 여부 - 일부 조직은 비공개를 선호
- SLA 페널티 조항 - 협력 관계에서는 불필요할 수 있음
- 아키텍처 선택권 제한 - 제안사의 창의적 제안 제한 우려
