## 3. RFP 1 (신규구축) 개선 사항

### 3.1 기대 효과 수치 근거 필요

**현재 (16줄)**:
> 물류 프로세스 자동화를 통한 작업 효율성 30% 이상 향상

**문제점**: 30% 수치의 근거가 없음. 업체가 이를 보증해야 하는지 불명확

**권장**:
- "작업 효율성 향상" 정도로 정성적 표현 사용
- 또는 벤치마크 대상 명시 (예: "현재 수작업 대비")

### 3.2 기술 스택 선택 범위가 너무 넓음

**현재 (575-579줄)**:
```
**백엔드**:
- 언어: Java, Kotlin, Python, Node.js 중 선택 (제안서에 명시)
```

**문제점**:
- 너무 다양한 선택지로 인해 비교 평가 어려움
- 유지보수 시 기술 역량 확보 부담

**권장**:
- 선호 기술 스택 명시 (예: "Java/Kotlin + Spring Boot 선호")
- 또는 RFP 2와의 일관성을 위해 Spring Boot 권장 명시

### 3.3 3PL 시스템 범위 모호

**현재 (512줄)**:
> **비고**: 3PL 시스템 기능 범위는 개발 산정 과정에서 변경될 수 있음

**문제점**:
- 견적 산출 시 업체가 범위를 특정하기 어려움
- 추후 분쟁 소지

**권장**:
- 최소 기능 범위(MVP)를 명확히 정의
- 선택적 기능은 별도 옵션으로 분리

### 3.4 첨부 자료 "별도 제공 예정" 다수

**현재 (773-774줄)**:
```
3. 자동화 설비 연동 사양 (별도 제공 예정)
4. 로케이션 구조 및 LPN 관리 정책 (별도 제공 예정)
```

**문제점**:
- 업체가 정확한 견적 산출 불가
- 제안서 품질 저하 가능성

**권장**:
- RFP 배포 전 해당 문서 완성
- 또는 제안 설명회에서 보충 설명

### 3.5 성능 요구사항 검증 방법 미명시

**현재 (600-604줄)**:
```
| **동시 사용자** | PC 웹 50명, PDA 100대 동시 접속 |
| **응답 시간** | 평균 1초 이내, 95% 3초 이내 |
| **처리량** | 일일 출고건 10,000건 이상 처리 |
```

**권장 추가**:
- 성능 테스트 방법 및 검증 기준 명시
- 부하 테스트 시나리오 제공

## 4. RFP 2 (운영유지보수) 개선 사항

### 4.1 운송사 계약주체 변경 대상 수 불일치

**RFP 2 (257줄)**:
```
**대상 운송사** (6개):
- DHL, UPS, FedEx, ACI, ARAMEX, DOORA, SF
```

**문제점**: 7개 운송사가 나열되어 있으나 "6개"로 표기

**권장**: "7개"로 수정하거나 목록 재확인

### 4.2 장애 대응 시간 현실성 검토

**현재 (106-111줄)**:
```
| **긴급** | 시스템 전체 중단 | 30분 이내 | 4시간 이내 |
| **높음** | 주요 기능 장애 | 1시간 이내 | 8시간 이내 |
```

**검토 필요**:
- 긴급 장애 30분 대응은 24시간 상주 인력 필요
- 이에 따른 비용 증가 고려 필요
- 업체 입장에서 SLA 페널티 리스크

**권장**:
- 운영 시간 명시 (평일 09:00~18:00 vs 24/7)
- 야간/휴일 긴급 장애 대응 별도 조건 명시

### 4.3 백업/복구 정책 상세화 필요

**현재 (168-171줄)**:
```
- RTO (Recovery Time Objective): 4시간
- RPO (Recovery Point Objective): 24시간
```

**검토 필요**:
- RPO 24시간은 하루치 데이터 손실 허용 의미
- 물류 시스템 특성상 출고 데이터 손실은 치명적

**권장**:
- RPO를 1시간 이내로 조정 검토
- 또는 "업무 크리티컬 데이터"와 "일반 데이터" 구분

### 4.4 추가 개발 우선순위 기준 모호

**현재 (483-488줄)**:
```
| **긴급** | 대체입고/출고, 지시재고이동, 운송사 계약주체 변경 | 3개월 |
| **높음** | 1주문 다출고, 로케이션 내 동일LPN 방지, 3PL 시스템 | 6~12개월 |
| **중** | 집품그룹 생성 개선, PDA 배정 개선... | 6~12개월 |
```

**문제점**:
- 긴급/높음/중/하의 분류 기준 불명확
- 비즈니스 임팩트, 기술 의존성 등 근거 미제시

**권장**: 우선순위 결정 기준 명시
- 예: 비즈니스 필요도, 기술 의존성, 사용자 요청 빈도 등

### 4.5 평가 기준 세부 항목 부재

**현재 (581-589줄)**:
```
| **시스템 이해도** | 20점 | GMS 시스템 분석, 운영 현황 파악 |
| **운영·유지보수 능력** | 25점 | 장애 대응 체계, SLA, 모니터링 방안 |
```

**권장**: 각 평가 항목별 세부 채점 기준 추가
- 예: 시스템 이해도 20점 = 기능 이해 10점 + 기술 스택 이해 5점 + 연동 시스템 이해 5점

## 7. 결론

두 RFP 문서는 GMS 시스템의 기능과 요구사항을 체계적으로 정리하고 있으며, 전반적인 구조와 내용은 양호합니다.

다만, 배포 전 다음 사항을 반드시 보완해야 합니다:

1. **[TBD] 항목 확정**: 마감일, 담당자, 계약조건 등
2. **민감 정보 제거**: 소스코드 저장소 경로
3. **수치 오류 수정**: 운송사 개수 (6개 → 7개)
4. **첨부 자료 완성**: 자동화 설비 연동 사양

위 사항이 보완되면 업체에 배포 가능한 수준의 RFP입니다.
