### 편집 1: 섹션 19에 새로운 하위 섹션 추가 (Kent Beck 인사이트 반영)

**위치**: 1745-1750번 라인 근처 (19.5 개인적 인사이트 섹션 내부)

**추가 내용**:
```markdown
#### AI 사용 전 "생각 먼저" 원칙: 전문가의 문제 접근법

> ⚠️ 핵심 통찰
> AI를 사용하기 전에 충분히 생각하고 AI를 사용해야지, 바로 AI를 사용하면 AI에게 끌려다닌다.

##### 왜 내 고민이 먼저인가?

AI가 제공하는 결과는 다른 사람들도 얻을 수 있는 **80% 수준**이다.
차별화된 경쟁력은 내 고민이 선행되어야 얻을 수 있다.

##### Kent Beck: 전문가들의 문제 접근 방법

> "전문가들은 마법사 같은 힘을 사용하지 않는다.
> **문제를 작은 문제로 분리해서 일반적인 수준의 기술로 처리할 수 있도록 한다.**
> 이런 분해는 많은 창의력과 기술을 요한다."
> — [[A Daily Practice of Empirical Software Design - Kent Beck - DDD Europe 2023]]

**전문가 vs 초보자:**

| 초보자 (4~5년차까지) | 전문가 |
|---------------------|--------|
| 문제를 열심히 푼다 | 문제를 **작게 분해**한다 |
| 복잡성을 직접 다룬다 | 복잡성을 **분리**한다 |
| 트릭과 기술을 축적 | 일반적 기술로 처리 가능하게 만듦 |
| 뇌가 포화될 때까지 확장 | **한번에 하나씩** 처리 |

**복잡성 분리 전략의 예시:**
- **행위(Behavior)와 구조(Structure) 분리**
- **DDD의 Bounded Context**
- **TDD**: "어떻게 구현할지 모르면 테스트를 먼저 추가. 구현은 쉽다. 하지만 함께 하면 힘들다."

> "한번에 하나씩 하면 뇌가 감당 가능. 함께 하면 과부하"
> — Kent Beck

##### AI 시대에 이것이 중요한 이유

```
❌ AI를 바로 사용하면:
   └── 문제 분해 과정을 건너뜀
   └── AI에게 끌려다님
   └── 80% 수준의 결과 (누구나 얻을 수 있는 것)

✅ 먼저 생각하면:
   └── 문제를 작은 문제로 분리
   └── 각 작은 문제에 AI 활용
   └── 차별화된 경쟁력 확보
```

**관련 개념들:**
- **CTA (Cognitive Task Analysis)**: AI가 잘 못하는 것은 사람의 고도화된 전문성과 암묵지가 요구되는 작업
- **STICC 프레임워크**: How보다 **What + Why**가 더 중요
- **1000x 차별화 능력**: 담대한 비전을 가지고, 비전을 향하는 마일스톤을 설정하고, 설계를 꾸준히 조정하는 능력

> "치열하게 고민하여 작업 과정을 쪼개면 AI에게 위임하기도 훨씬 수월해진다"
> — [[나날이 발전하고픈 개발자를 위한 AI 활용법]]

##### 실천 방법

1. **문제 분해 먼저**: AI에게 맡기기 전에 문제를 작은 문제로 분리
2. **입력-출력 기대치 정의**: 각 작은 문제의 예상 결과 명확화
3. **내가 직접 한다면?**: 어떤 순서로 할지 생각 (CTA)
4. **검증 기준 수립**: AI 결과물의 품질 판단 기준
5. **불확실성 지도**: AI의 한계 파악

**관련 문서**:
- [[A Daily Practice of Empirical Software Design - Kent Beck - DDD Europe 2023]]
- [[나날이 발전하고픈 개발자를 위한 AI 활용법]]
- [[AI-Assisted Development With Verification (AADV)]]
```

### 편집 2: 섹션 12에 Vibe Coding 팀 적용 한계 추가 (AI Paradox 반영)

**위치**: 기존 "Vibe Coding의 한계와 검증 필수" 섹션 (문서 중간부)

**추가 내용**:
```markdown
#### 개인 vs 팀 프로젝트: AI Paradox와 Vibe Coding의 적용 범위

> ⚠️ 중요 구분
> 개인 과제는 Vibe Coding이 가능하지만, **팀 프로젝트에서는 불가능하다**.

##### AI Paradox: 더 빠른 코딩, 더 느린 배포

Addy Osmani(Google Chrome DevTools)가 발표한 **AI Paradox** 데이터:

| 지표 | 개인 수준 | 팀 수준 |
|------|----------|---------|
| 태스크 완료 | **+21%** 증가 | - |
| PR 리뷰 시간 | - | **+91%** 증가 |
| PR 크기 | - | **+154%** 증가 |

> "시스템은 가장 느린 단계의 속도로 움직인다. 오늘날 그 단계는 **리뷰와 검증**이다."
> — [[AI-Paradox-Faster-Coding-Slower-Shipping-Addy-Osmani]]

##### 70% 문제: 데모 vs Production

```
AI의 70%: 스캐폴딩, 기본 기능, "뭐, 작동은 하네"
인간의 30%:
├── 엣지 케이스
├── 아키텍처 문제
├── 테스트
├── 클린업
└── Production Readiness ← 가장 어려운 부분
```

**팀에서의 문제점:**
- 주니어가 그 70%를 **완료된 것처럼** PR로 제출
- 실제 어려운 부분이 **코드 리뷰로 전가**
- 시니어가 **인간이라면 만들지 않았을 실수**를 정리하게 됨

##### Vibe Coding vs AI Assisted Engineering

| Vibe Coding | AI Assisted Engineering |
|-------------|------------------------|
| 빠르고 탐색적 | 체계적 통합 |
| 속도와 실험 우선 | **인간이 통제권** 유지 |
| **깊은 리뷰 후순위** | **구조화된 프로세스** |
| 개인 프로젝트에 적합 | **팀 프로젝트 필수** |

##### 팀에서의 스킬 침식 방지 전략

**1. 트리오 프로그래밍**:
```
시니어 + 주니어 + AI
├── 시니어: 전문가가 루프에 유지
├── 주니어: AI를 활용한 속도 레버리지
└── 품질: 협업적 인간-AI 팀워크
```

**2. 소크라테스식 코드 리뷰**:
```
주니어가 AI 생성 PR 제출
    ↓
시니어가 소크라테스식 질문
    ↓
AI의 출력을 설명하도록 강제
    ↓
기능성이 아닌 **이해도** 확보
```

**3. No-AI 챌린지**: 정기적으로 AI 없이 코딩하여 비판적 사고 스킬 유지

##### 팀 프로젝트에서 필요한 것

1. **테스트**: AI 코딩의 안전망. 탈선 시 더 빨리 알 수 있음
2. **CI/CD**: 자동화된 검증 게이트
3. **명확한 문서화**: 컨텍스트 엔지니어링의 기초
4. **AADV 프로세스**: Create → Assess → Execute 사이클

> "AI는 더 많은 코드를 빠르게 작성하는 것이 아니라, **더 나은 소프트웨어를 만드는 것**이다."
> — Addy Osmani
```
